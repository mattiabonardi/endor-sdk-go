# Internazionalizzazione (i18n)

Il package `sdk_i18n` fornisce un sistema di traduzione basato su file YAML con supporto a placeholder nominali e fallback automatico alla lingua di default.

---

## Inizializzazione

Prima di usare qualsiasi funzione di traduzione è necessario chiamare `sdk_i18n.Init` passando il percorso alla directory delle traduzioni del progetto:

```go
import "github.com/mattiabonardi/endor-sdk-go/pkg/sdk_i18n"

func main() {
    if err := sdk_i18n.Init("./locales"); err != nil {
        log.Fatal(err)
    }
    // ...
}
```

Se la directory non esiste o il percorso è vuoto, l'SDK si avvia comunque usando le sole traduzioni embedded (vedere la sezione [Fallback](#catena-di-fallback)).

---

## Struttura dei file di traduzione

I file di traduzione sono file YAML con chiavi annidate. Le chiavi vengono appiattite usando la notazione a punti.

**Struttura directory consigliata:**

```
./locales/
    en.yaml
    it.yaml
    fr.yaml
```

**Esempio `locales/en.yaml`:**

```yaml
commons:
  not_found: "Page not found (uri: {{uri}}, method: {{method}})"

orders:
  created: "Order {{id}} created successfully"
  not_found: "Order {{id}} not found"

errors:
  generic: "An unexpected error occurred"
```

Il nome del file determina la locale (es. `en.yaml` → locale `en`).

### Chiavi flatten

Il YAML annidato viene letto come chiavi separate da punti:

| YAML annidato                | Chiave flat              |
|------------------------------|--------------------------|
| `commons.not_found`          | `commons.not_found`      |
| `orders.created`             | `orders.created`         |

## Funzione di traduzione

```go
func T(locale, key string, args map[string]any) string
```

Traduce `key` per la `locale` specificata con interpolazione di placeholder nominali nella forma `{{nome}}`.

| Parametro | Tipo             | Descrizione                                            |
|-----------|------------------|--------------------------------------------------------|
| `locale`  | `string`         | Codice lingua (es. `"en"`, `"it"`)                    |
| `key`     | `string`         | Chiave dot-notation (es. `"orders.not_found"`)         |
| `args`    | `map[string]any` | Valori per i placeholder `{{key}}`. `nil` = nessuna interpolazione |

**Esempi:**

```go
// Senza placeholder
msg := sdk_i18n.T("en", "errors.generic", nil)
// → "An unexpected error occurred"

// Con placeholder
msg := sdk_i18n.T("en", "orders.not_found", map[string]any{"id": "42"})
// → "Order 42 not found"

// Chiave inesistente: viene restituita la chiave stessa
msg := sdk_i18n.T("en", "this.key.does.not.exist", nil)
// → "this.key.does.not.exist"
```

### NormalizeLocale

```go
func NormalizeLocale(acceptLanguage string) string
```

Estrae il tag lingua primario dall'header HTTP `Accept-Language`. Usato internamente dall'SDK per risolvere la locale dalla richiesta.

```go
sdk_i18n.NormalizeLocale("it-IT,it;q=0.9,en;q=0.8") // → "it"
sdk_i18n.NormalizeLocale("en-US")                    // → "en"
sdk_i18n.NormalizeLocale("")                         // → "en"
```

## Catena di fallback

Quando si cerca una chiave per la locale `"it"`, l'SDK segue questo ordine:

```
1. Traduzioni del progetto  →  locales/it.yaml
2. Traduzioni SDK embedded  →  sdk/locales/it.yaml
3. Traduzioni del progetto  →  locales/en.yaml   (DefaultLocale)
4. Traduzioni SDK embedded  →  sdk/locales/en.yaml
5. La chiave stessa         →  "orders.not_found"
```

Questo permette al progetto di sovrascrivere selettivamente le traduzioni dell'SDK aggiungendo le stesse chiavi nel proprio file YAML senza dover ridefinire tutto.

## Traduzione nei handler — `EndorContext.T`

All'interno di un handler l'`EndorContext` espone direttamente la funzione `T` con la locale già risolta dall'header `Accept-Language` della richiesta:

```go
func (ec *EndorContext[T]) T(key string, args map[string]any) string
```

**Esempio:**

```go
func myHandler(c *sdk.EndorContext[MyPayload]) (*sdk.Response[MyResult], *sdk.EndorError) {
    msg := c.T("orders.created", map[string]any{"id": "99"})
    // msg → "Order 99 created successfully"
    return sdk.NewResponseBuilder[MyResult]().
        AddMessage(sdk.NewMessage(sdk.ResponseMessageGravityInfo, msg)).
        Build(), nil
}
```

## Errori con traduzione — `EndorError`

`EndorError` supporta la traduzione lazy tramite `WithTranslation`. La traduzione viene risolta al momento della risposta HTTP usando la locale della richiesta, quindi **non è necessario conoscere la locale al momento in cui si crea l'errore**.

```go
func (e *EndorError) WithTranslation(key string, args map[string]any) *EndorError
```

**Esempio:**

```go
func myHandler(c *sdk.EndorContext[MyPayload]) (*sdk.Response[MyResult], *sdk.EndorError) {
    order, err := repo.FindByID(c.Payload.ID)
    if err != nil {
        return nil, sdk.NewNotFoundError(err).
            WithTranslation("orders.not_found", map[string]any{"id": c.Payload.ID})
    }
    // ...
}
```

Le factory di errore disponibili sono:

| Factory                        | HTTP Status |
|--------------------------------|-------------|
| `NewBadRequestError(err)`      | 400         |
| `NewNotFoundError(err)`        | 404         |
| `NewConflictError(err)`        | 409         |
| `NewForbiddenError(err)`       | 403         |
| `NewUnauthorizedError(err)`    | 401         |
| `NewInternalServerError(err)`  | 500         |
| `NewGenericError(status, err)` | custom      |

## Tag `schema` sui modelli

Lo schema JSON di un modello Go viene generato automaticamente tramite reflection. Ogni campo dello struct può essere annotato con i tag `schema` e `ui-schema` per arricchire la descrizione del campo.

### Tag `schema`

```go
FieldName Type `json:"field_name" schema:"key=value,key2=value2"`
```

| Chiave        | Tipo     | Descrizione                                                                 |
|---------------|----------|-----------------------------------------------------------------------------|
| `title`       | string   | Etichetta del campo. Supporta la sintassi `t(key)` per la traduzione i18n  |
| `description` | string   | Descrizione estesa. Supporta la sintassi `t(key)` per la traduzione i18n   |
| `format`      | string   | Formato OpenAPI/JSON Schema (es. `date-time`, `email`, `uuid`, `yaml`, …)  |
| `readOnly`    | `true`   | Il campo è di sola lettura                                                 |
| `writeOnly`   | `true`   | Il campo è di sola scrittura                                               |
| `minLength`   | intero   | Lunghezza minima (campi stringa)                                            |
| `maxLength`   | intero   | Lunghezza massima (campi stringa)                                           |
| `enum`        | string   | Valori ammessi separati da `\|` (es. `enum=active\|inactive`)               |
| `uniqueItems` | `true`   | Elementi univoci in un array                                                |

**Formati disponibili (`format`):**

`date-time` · `date` · `time` · `email` · `hostname` · `ipv4` · `ipv6` · `uri` · `uuid` · `password` · `country-code` · `language-code` · `currency` · `yaml` · `json` · `asset` · `image-asset` · `audio-asset` · `video-asset`

### Traduzione di `title` e `description` con `t(key)`

I valori di `title` e `description` possono essere statici oppure contenere la sintassi `t(key)` per richiedere una traduzione dinamica. Quando il framework genera lo schema da inviare al client, chiama `RootSchema.ResolveTranslations(locale)` che sostituisce ogni token `t(key)` con il valore tradotto nella lingua della richiesta.

```go
type BaseHandlerModel struct {
    ID        string `json:"id"        schema:"title=Id,readOnly=true"`
    Attribute string `json:"attribute" schema:"title=t(commons.data)"`
}
```

- `title=Id` — valore statico, viene usato così com'è
- `title=t(commons.data)` — il framework risolve la chiave `commons.data` nel file di traduzione della locale corrente

Il file di traduzione corrispondente:

```yaml
# locales/en.yaml
commons:
  data: "Data"

# locales/it.yaml
commons:
  data: "Dato"
```

Lo stesso meccanismo vale per `description`:

```go
Price float64 `json:"price" schema:"title=t(products.fields.price),description=t(products.fields.price_desc)"`
```

### Tag `ui-schema`

Il tag `ui-schema` aggiunge metadati UI usati dal frontend. Può essere applicato sia a livello di campo sia a livello di struct.

**A livello di campo:**

| Chiave        | Tipo   | Descrizione                                                                 |
|---------------|--------|-----------------------------------------------------------------------------|
| `entity`      | string | Entità di riferimento per i campi a selezione dinamica                     |
| `query`       | string | Query per recuperare i dati dell'entità referenziata (`$filter() $projection()`) |
| `hidden`      | `true` | Nasconde il campo nell'interfaccia utente                                  |

**A livello di struct (root):**

| Chiave                 | Tipo   | Descrizione                                                              |
|------------------------|--------|--------------------------------------------------------------------------|
| `entityIdKey`          | `true` | Indica quale campo è la chiave ID dell'entità                            |
| `entityDescriptionKey` | `true` | Indica quale campo rappresenta la descrizione dell'entità                |

**Esempio completo:**

```go
type Order struct {
    ID          string `json:"id"          schema:"title=t(orders.fields.id),readOnly=true"     ui-schema:"entityIdKey=true"`
    Description string `json:"description" schema:"title=t(orders.fields.description)"           ui-schema:"entityDescriptionKey=true"`
    Status      string `json:"status"      schema:"title=t(orders.fields.status),enum=open|closed|cancelled"`
    CustomerID  string `json:"customerId"  schema:"title=t(orders.fields.customer)"               ui-schema:"entity=customers,query=$filter(active==true) $projection(id,name)"`
    Notes       string `json:"notes"       schema:"title=t(orders.fields.notes)"                  ui-schema:"hidden=true"`
    CreatedAt   string `json:"createdAt"   schema:"title=t(orders.fields.created_at),format=date-time,readOnly=true"`
}
```

## Override delle traduzioni SDK

Il progetto può sovrascrivere qualsiasi chiave delle traduzioni SDK aggiungendo la stessa chiave nel proprio file YAML. Le chiavi SDK usate internamente sono:

```
commons.not_found
entities.entity.created
entities.entity.updated
entities.entity.deleted
entities.entity.not_found
entities.entity.already_exists
entities.entity.create_not_permitted
entities.entity.update_not_permitted
entities.entity.delete_not_permitted
entities.entity.action_not_found
entities.entity.invalid_action_id
entities.entity.created_specialized
entities.entity.updated_category
entities.aggregation.failed
```

**Esempio override `locales/en.yaml`:**

```yaml
entities:
  entity:
    not_found: "The requested resource with id {{id}} could not be found"
```
